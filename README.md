# Local RAG

Live demo: https://rag.maximilianholm.se/

A fully local Retrieval-Augmented Generation (RAG) application built with **FastAPI**, **PostgreSQL (TimescaleDB + pgvector)**, and **Ollama**. Documents are indexed into a vector database, semantically searched using `pgvector`, and used to augment responses generated by a local large language model. Every component runs on your own machine, ensuring that documents, queries, embeddings, and generated responses never leave your local environment.

## Overview

Large language models are capable of generating fluent, natural responses, but they are limited by the data they were trained on. When asked about information outside their training data, they may produce inaccurate or fabricated answers.

Retrieval-Augmented Generation (RAG) addresses this problem by retrieving relevant information from your own document collection before generating a response. Instead of relying solely on the model's internal knowledge, the model is provided with context retrieved from documents that you have ingested.

This project keeps the entire RAG pipeline local. Documents are embedded using Ollama, stored in PostgreSQL with `pgvector`, retrieved using vector similarity search, and finally passed back to Ollama to generate an answer. No external APIs are required after the models have been downloaded.

## Features

This project provides a complete local RAG pipeline including:

* Local document ingestion from files, folders, raw text, or URLs
* Automatic document chunking with overlapping context
* Vector embeddings generated locally using Ollama
* Semantic similarity search using PostgreSQL and `pgvector`
* Local response generation using Mistral through Ollama
* FastAPI backend for ingestion and querying
* Simple web chat interface for testing
* Optional Open WebUI integration
* Docker Compose deployment
* NVIDIA GPU support (optional)

## Architecture

The application consists of four main components.

**FastAPI** acts as the orchestration layer. It accepts uploaded documents, extracts their text, generates embeddings through Ollama, stores them in PostgreSQL, retrieves relevant document chunks during queries, and sends those results back to Ollama for answer generation.

**PostgreSQL (TimescaleDB)** stores every document chunk together with its vector embedding. The `pgvector` extension provides efficient vector storage and similarity search using cosine distance.

**Ollama** hosts both the embedding model and the language model locally. The embedding model converts documents and user queries into vector representations, while the language model generates responses using the retrieved document context.

The **Web Chat** provides a lightweight interface for uploading documents and interacting with the RAG system.

Everything communicates over the local Docker network, meaning no document content or prompts are sent to external services.

## How it works

The retrieval pipeline consists of five stages.

### 1. Document ingestion

Documents are uploaded through the web interface or the ingestion API. The FastAPI application extracts the document text before splitting it into overlapping chunks. Chunking improves retrieval quality by ensuring that relevant information is preserved across chunk boundaries.

### 2. Embedding generation

Each document chunk is sent to Ollama's `nomic-embed-text` embedding model through the Ollama REST API. The model converts every chunk into a high-dimensional vector representing its semantic meaning.

Those vectors are stored alongside the original text inside PostgreSQL using the `pgvector` extension.

### 3. Semantic retrieval

When a user submits a question, the query is embedded using the same embedding model.

PostgreSQL compares the query embedding against every stored document embedding using cosine similarity (`<=>`) and returns the most relevant document chunks.

### 4. Response generation

The retrieved document chunks are combined into a prompt together with the user's question.

This prompt is sent to the local Mistral model hosted by Ollama. The language model is instructed to answer only using the retrieved context and to avoid generating information that is not present in the supplied documents.

### 5. Return the response

The generated answer, together with the retrieved document sources, is returned to the web interface.

## Quick Start

Clone the repository and create an environment file.

```bash
cp .env.example .env
```

Open the `.env` file and configure a strong value for:

```text
INGEST_API_KEY
```

Build and start the stack.

```bash
docker compose up --build -d
```

On first startup, Ollama automatically downloads the required models into a persistent Docker volume.

Once the containers have started, open:

| Service | URL                    |
| ------- | ---------------------- |
| RAG API | http://localhost:8044  |
| Ollama  | http://localhost:11434 |

If these ports are already in use, they can be changed in `docker-compose.yml`.

## Ingesting Documents

Documents can be uploaded directly through the web interface using the **Upload Files/Folders** button.

The backend accepts PDF documents, text files, raw text, and URLs. During ingestion the document is extracted, chunked, embedded, and stored in PostgreSQL.

You can also ingest documents using the API.

```bash
curl -X POST http://localhost:8044/ingest \
  -H "X-API-Key: $INGEST_API_KEY" \
  -F "file=@/path/to/document.pdf"
```

Or submit raw text.

```bash
curl -X POST http://localhost:8044/ingest \
  -H "X-API-Key: $INGEST_API_KEY" \
  -F "text=This is a sample document."
```

## Querying

Once documents have been ingested, queries can be sent to the API.

```bash
curl -X POST http://localhost:8044/query \
  -H "Content-Type: application/json" \
  -d '{
        "query": "What is contained in the uploaded documents?"
      }'
```

The application embeds the query, retrieves the most relevant document chunks, constructs a prompt, generates an answer using Ollama, and returns both the answer and the retrieved sources.

## Optional: Open WebUI

Open WebUI provides a more fully featured interface for interacting with the same Ollama models.

Start it using the compose override.

```bash
docker compose -f docker-compose.yml -f docker-compose.openwebui.yml up --build -d
```

Then open:

```
http://localhost:8080
```

Create an administrator account when prompted and select one of the installed Ollama models.

## GPU Support

If an NVIDIA GPU is available together with the NVIDIA Container Toolkit, Ollama can run using GPU acceleration.

Start the stack using the GPU override.

```bash
docker compose -f docker-compose.yml -f docker-compose.gpu.yml up --build -d
```

To verify GPU usage:

```bash
docker logs ollama | grep -i gpu
```

Without the GPU override, Ollama runs entirely on the CPU.

## Some reflections

### Cosine similarity score (threshold)
I have experimented with different thresholds for the cosine difference used in the semantic filtering step. The threshold has a surprisingly large impact on how the RAG system behaves. For example, using a loose threshold (ΔS < 0.60) makes the system almost always find something to respond with, but this also increases the risk of retrieving irrelevant information. On the other hand, a stricter threshold (ΔS < 0.40) makes the chatbot extremely conservative, almost like a document search engine, where responses are limited to exactly what exists in the retrieved chunks.

From my experiments, a threshold somewhere between 0.45 and 0.55 seems to provide a good balance for my specific use case, which is a personal portfolio chatbot that answers questions about me based on my own documents.

However, one important observation is that cosine similarity is not the same as answer quality. A high similarity score does not necessarily mean that a chunk contains the correct answer. Embeddings are good at measuring semantic closeness, but they do not truly understand context, intent, or whether the retrieved information is useful for generating a correct response. A chunk can be semantically similar while still being irrelevant or incomplete.

Therefore, cosine similarity should be viewed as a first-stage filtering mechanism rather than a final decision-maker. A better RAG pipeline would likely combine embedding similarity with additional methods such as reranking models, keyword-based matching, metadata filtering, or evaluation of the generated answer itself.

For this project, cosine similarity worked well enough because the knowledge base was relatively small and focused. However, for larger or more complex RAG systems, relying only on cosine similarity could lead to incorrect retrievals and hallucinated answers.

### Get a nice GPU for the love of god

Finally, I would strongly recommend running the system with a GPU whenever possible. My experiments were performed using a relatively weak GPU (GTX 1060 6GB), and even with this hardware the response time was significantly better compared to CPU-only. Running the model on CPU made the response generation noticeably slow and would not be a practical solution for an interactive chatbot.
